Why & How It Works

🗺️ What you're about to do

You're going to use Reinforcement Fine-Tuning (RFT) to teach a small AI model to get better at math. By the end, Qwen3-0.6B will go from ~40% accuracy on grade-school math problems to 80%+ — all through trial, error, and reward signals.

The pipeline in plain English

1. The model reads a math problem and generates an answer
2. An evaluator checks if the answer is right (reward = 1) or wrong (reward = 0)
3. The model updates its weights to produce more rewarded answers
4. Repeat 1,000 problems × 3 epochs = a smarter model

🧰 Tools you'll install

GSM8K dataset 1,000 grade-school math problems

🍺 What is Homebrew?

Homebrew is a package manager — think of it like an App Store for developer tools, but run from the terminal. It handles downloading, installing, and updating software like Python, Git, and more.

🏪 Analogy

Without Homebrew, installing Python 3.11 would mean going to python.org, downloading an installer, clicking through a wizard, hoping it works. With Homebrew, it's one command: brew install python@3.11.

Why does Mac need this?

macOS ships with Python 3.9 (old and locked down by Apple) and no easy way to upgrade it. Homebrew gives you a separate, manageable Python installation that doesn't interfere with Apple's system Python.

🛤️ What is PATH?

When you type a command in Terminal (like brew), your Mac doesn't search every folder — it only looks in specific folders listed in your PATH. Homebrew installs itself in /opt/homebrew/bin which isn't in PATH by default.

📬 Analogy

Imagine your terminal is a mail carrier that only delivers to streets on a specific map. Homebrew built a new street (/opt/homebrew/bin) but the mail carrier's map doesn't include it yet. These 3 commands update the map.

What is .zprofile?

It's a config file that runs automatically every time you open a new terminal window. By adding Homebrew there, you never have to think about this again — it'll always be available.

🐍 Why Python 3.11 specifically?

The eval-protocol package requires Python 3.10 or newer. Your Mac ships with 3.9, which is close but not enough. Python 3.11 is currently the most stable modern version.

Python version landscape

🔴 Python 3.9 — Mac system default, too old
🟡 Python 3.10 — minimum required
🟢 Python 3.11 — what we install, recommended
🟢 Python 3.12/3.13 — also fine if you have them

Why pip3.11 instead of pip?

Because you now have multiple Pythons. Using pip3.11 ensures packages install into the right Python. Using just pip might accidentally use the old system Python and packages won't be found.

📦 What is eval-protocol?

eval-protocol is Fireworks' open-source CLI that bridges your local evaluator code with their cloud training infrastructure. It handles uploading your dataset, registering your scoring function, and submitting the RFT job.

📚 What is the GSM8K dataset?

GSM8K (Grade School Math 8K) is a benchmark of ~8,500 grade-school math word problems created by OpenAI. Each problem requires multi-step reasoning — not just arithmetic, but understanding what to calculate and in what order.

📝 Example problem

"Janet has 3 cats. She buys 4 more. Then gives 2 away. Her friend gives her 1 back. How many cats does Janet have?" → The model must track state across multiple steps to get 6.

🐛 Why does this bug exist?

The eval-protocol CLI registers many commands, and two of them both tried to use --credentials-secret as a short flag. Python's argparse library (which handles CLI arguments) raises an error when two flags share the same name.

What the patch does

It wraps the parser.add_argument() calls in a try/except. When the second flag tries to register and finds a conflict, the exception is caught and silently skipped — the first registration wins, which is fine.

Is this safe?

Yes — we're only suppressing a duplicate registration error, not hiding actual runtime failures. The CLI still works correctly; we're just letting it load without crashing on startup.

🧪 What does eval-protocol local-test actually do?

It runs the math evaluator against a few sample problems using your current model (or a test model). This verifies your evaluator logic works before spending compute on training.

Why does it also upload things?

Fireworks needs two things registered before it can train: your evaluator (the scoring function) and your dataset (the 1,000 problems). Running the test locally also handles this registration automatically — one command, two jobs done.

What is a rollout?

Each time the model answers a question during training, that's called a rollout. The 1,600 "Successful Rollouts" you see in the dashboard = 200 problems × 8 response candidates per problem.

🤖 What is RFT vs normal fine-tuning?

Supervised fine-tuning (SFT) — show the model correct answers and train it to copy them. Simple but requires lots of labeled examples.

Reinforcement fine-tuning (RFT) — let the model generate its own answers, score them, reward the good ones. The model learns through trial and error, not imitation.

The algorithm: GRPO

Fireworks uses Group Relative Policy Optimization. For each question, it generates 8 answer candidates, scores all 8, then asks: "which answers were better than average?" Those get positive gradient, the rest get negative. No absolute reward needed — just relative comparison.

Why binary rewards still work

You might think RL needs a smooth reward spectrum (-1 to 1). But binary (0/1) is fine — it's just sparse. GRPO compensates by comparing 8 candidates against each other, so even "all wrong" gives signal: the least-wrong answer gets relative credit.

📊 Understanding the config

LoRA Rank: 8

Instead of retraining the whole model, LoRA adds tiny "adapter" matrices (rank 8) alongside frozen original weights. If a weight matrix is 1024×1024 (1M params), LoRA only trains two matrices of size 1024×8 and 8×1024 = 16K params. 63x fewer updates, similar quality.

Response Candidates: 8

For each math problem, the model generates 8 different answers. All 8 get scored. GRPO compares them to each other to compute the gradient signal. More candidates = better signal but more compute.

Temperature: 1.0

Full randomness during candidate generation. You want diversity across the 8 candidates so the model explores different reasoning strategies. Low temperature would make all 8 nearly identical — useless for RL.

Chunk Size: 200 rows

The 1,000 problems are split into 5 chunks of 200. Each chunk trains sequentially. You can inspect scores per-chunk in the dashboard Details tab.

📈 Reading the score histogram

You'll see two tall bars — one near 0.0 and one near 1.0. This "bimodal" distribution is completely normal for binary RL. The model either gets math problems right (1.0) or wrong (0.0) — no partial credit.

What to watch for over epochs

As training progresses, the right bar (1.0) should grow and the left bar (0.0) should shrink. The Average Score climbs from ~0.4 baseline to 0.75–0.85 after 3 epochs. That's your model learning math.

What happens after training?

Your fine-tuned model is saved as a Fireworks-hosted endpoint. Click Deploy in the dashboard to make it available via API — same OpenAI-compatible format you'd use to call any other model. You can then query it with math problems and compare accuracy to the base Qwen3-0.6B.

🎓 What you actually built

A specialist — a model that started as a generalist (knows language, facts, reasoning) and was RL-trained to excel specifically at grade-school math. Same principle companies like Vercel use to fine-tune models for code generation.
